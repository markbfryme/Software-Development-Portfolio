  • Lecture 54: Handling Events
    • event handlers are basically functions that you bind using specific methods to certain events that happen in the browser
    • placing an on..something in the html element is one option
    • There are consequences, mainly it "dirties" up your html code
    • There is another way called unobstrusive event binding in which the html doesn't need to know anything about it.
    • 
          document.querySelector("button")
            .addEventListener("click", sayHello);
    • There is a similar way to do this with the on..something method
    • 
          document.querySelector("button")
            .onclick = sayHello;
    • This gives more flexibility
    • You can make the name of the button change when it is clicked easily because the this object when the function is exectuted is now pointing to the right element
    • you can also use this to listen for the event DOMContentLoaded so that the script.js can be referenced in the head rather than the end of the document

  • Lesson 56: HTTP Basics
    • Language in which browser talks to the server
    • HTTP: HyperText Transfer Protocal
    • Based on request/response stateless protocol
    • Client (browser) opens a connection to a server
    • Client sends HTTP request for a resource
    • Server sends HTTP response to the client (w/resource)
    • Client closes connection to server
    • URN:
      • Uniform Resource Name
      • Uniquely identifies resource or name of resource
      • Does not tell us how to get the resource
      • Maintain unique names of courses with string from identifiers
      • example: "HTML/CSS/Javascript/Web Developers/Yaakov/Chaikin"
    • URI
      • Uniquely identifies resource of location of resource
      • Does not necessarily tell us how to get the resource
      • Still doesn't tell us how/where to get the resource
      • example: /official_web_site/index.html
    • URL:
      • Form of URI that provides info on how to get resources
      • example: http://www.mysite.com/official_web_site/index.html
    • HTTP Request Structure (GET)
      • GET /index.html?firstName=Yaakov HTTP/1.1
      • GET is the Method
      • /index.html is the URI String
      • firstName=Yaakov is the Query String
        • name/value pairs separated by & Example: ?first=Yaakov&last=Chaikin
      • HTTP/1.1 is the HTTP version
      • There's something strange about this command, its a URI not a URL
      • only issued after we are connected to a resource (website)
      • We are now asking with the protocol HTTP after we are already connected to the server
      • This is a GET command or request
    • HTTP Methods
      • GET
        • Retrieves the resource
        • Data is passed to server as part of the URI ie query string
      • POST
        • Sends data to server in order to be processed
        • Data is sent in the message body
      • Many other methods not covered here
    • HTTP Request Structure (POST) 
      • 
              POST /index.html HTTP/1.1
              Host: coursera.org
              Accept-Charset: utf-8
              firstName=Yaakov...
              ...
              ...
    • HTTP Response Structure 
      • 
              HTTP/1.1 200 OK
              Date: Tue, 11 Aug 2004 19:00:01 GMT
              Content-Type: text/html
              ... html page here ...
              ...
              ...
      • HTTP Version Response, status code, English phrase describing status code all seperated by a single space
      • 200 OK: Ok, here is the content you requested
      • 404 Not Found: Server can't find the resource requested
      • 403 Forbidden: Unauthenticated client tried to access a secure resource
      • 500 Internal Server Error: Some unhandled error was raised on the server
      • If you are going to be doing client server communication you need to understand the basics of HTTP protocol
      • GET method sends data to server as part of the URL
      • POST method sends data as part of message body
      • Common Response codes: 200, 404, 403, 500, etc.

  • Lecture 57, Part 1: AJAX Basics
    • AJAX stands for Asynchronous Javascript and XML
      • While AJAX started with XML, very few apps use XML with AJAX today
      • plain text (at times as html) or JSON is used instead
    • Why AJAX? It allows small pieces of data to be requested with Javascript rather than requesting and receiving a full yet slightly modified webpage.
    • Faster Response, less bandwidth, nicer user experience
    • Synchronous: The execution of one programming instruction at a time
    • Can't start the execution of another until first is exectued
    • Asynchronous: Execution of more than one instruction at a time.
    • Asynchrounous instruction returns right away
    • The actual execution is done in a separate thread or process
    • If Javascript is synchronous how does AJAX work?
      • The Javascript engine in a web browser is synchronous
      • The HTTP Requestor is Asynchronous
      • Note: There are other components in a web broswer, Event Queue, webGL, HTML Rendering
      • Use a special object to make an AJAX request from the HTTP Requestor
      • At the same time we send a response handler function
      • Javascript doesn't wait on the response and keeps synchronously executing
      • Then a callback function is used to get the response from the HTTP Requestor...

    • window.XMLHttpRequest is the most current request object, the others are very unlikely to be in use

    • the main function we care about is attached to ajaxUtils, ajaxUtils.sendGetRequest
    • arguments are requestUrl and responseHandler
    • first gets the request object

    • set request.onreadystatechange to another function handleResponse(request, responseHandler)
    • We're not finished setting up the request
    • Set the open method request.open("GET", requestURL, true); the true makes it asynchronous
    • The nest line request.send(null) actually finally sends the request... everything else before this point was just setting up the request
    • The handleResponse function takes in the request and the responseHandler and makes sure we are in the request.readyState == 4, the last state, then we check if the request.status == 200. If those two conditions are true, we pass the request to the responseHandler... responseHandler(request)
    • We then expose the utility to the global object. global.$ajaxUtils = ajaxUtils;
